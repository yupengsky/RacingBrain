# 亮点、问题与改进路线

## 1. 工程亮点

### 1.1 清晰的多传感器分工

工程没有让单一传感器承担全部任务，而是明确分工：

```text
Camera -> 颜色语义
LiDAR  -> 空间位置
GNSS/INS -> 车辆全局位姿
SLAM -> 时序稳定和全局地图
```

这种分工符合无人方程式场景。

### 1.2 完整 ROS 2 端到端链路

工程不是孤立算法脚本，而是完整 ROS 2 工作空间：

- 自定义消息。
- C++ 节点。
- Python 节点。
- launch 集成。
- RViz 可视化。
- rosbag 回放验证。
- 一键构建和运行脚本。

### 1.3 融合层有实际工程容错

标准融合使用 3D 投影 2D IoU 匹配。同时又增加了：

- Unknown 保留。
- 单目补漏。
- 中心距离强制上色。

这些策略说明工程考虑到了真实传感器误差和标定误差。

### 1.4 SLAM 后端不是简单累积点

SLAM 后端有完整状态管理：

- `GlobalCone` 地图对象。
- 协方差 `P`。
- 颜色门控。
- 欧氏距离粗筛。
- 马氏距离匹配。
- 动态观测噪声。
- Kalman 更新。
- hit/miss 存在评分。
- 稳定发布。

这是工程最有技术价值的部分。

### 1.5 适配赛车比赛节奏

循迹模式实现了：

```text
探索圈建图
回环检测
删除不稳定锥桶
锁图
冲刺圈复用地图
```

这比单纯一直增量建图更适合比赛。

## 2. 当前问题

### 2.1 skidpad 八字模式未实现

`processSkidpadTrack` 当前是 TODO。

影响：

- 八字绕环无法作为完整功能汇报。
- 可以说“已预留模式接口和配置，待实现专用策略”。

### 2.2 `/global_map` 是 MarkerArray

当前 SLAM 最终输出：

```text
visualization_msgs/msg/MarkerArray
```

它适合 RViz，不适合作为规划算法的稳定数据接口。

建议增加：

```text
/global_cone_map    drd25_msgs/msg/Map
```

或者更完整的全局地图消息，包含：

- `id`
- `x`
- `y`
- `color`
- `confidence`
- `covariance`

### 2.3 Unknown 锥桶被 SLAM 跳过

融合层会保留 Unknown，但 SLAM 中：

```cpp
if (cone_obs.color == UNKNOWN)
    continue;
```

影响：

- LiDAR 检测到但相机没识别颜色的锥桶完全不入图。
- 召回率受相机颜色识别影响较大。

改进：

- 可以允许 Unknown 作为低权重候选入图。
- 后续被相机识别到颜色后再更新 type。

### 2.4 force_match_radius 是经验策略

当前默认 60 像素，作用是外参误差下强制上色。

风险：

- 锥桶密集时可能错配。
- 参数对相机分辨率敏感。

改进：

- 用匈牙利匹配做全局最优关联。
- 同时考虑 IoU、中心距离、深度排序。
- 根据 bbox 大小自适应阈值。

### 2.5 畸变参数读取但未用于投影

融合层读取了：

```text
dist_coeffs
```

但投影函数中手写针孔模型，没有使用 `cv::projectPoints` 或 `undistortPoints`。

改进：

- 使用 OpenCV 投影函数。
- 或先对图像点做畸变校正。

### 2.6 gyro_factor 参数未实际使用

SLAM 读取了：

```text
gyro_factor
```

但锁图后的 yaw rate 惩罚代码使用：

```cpp
params_.speed_factor * std::abs(current_gyro_z_)
```

可能应为：

```cpp
params_.gyro_factor * std::abs(current_gyro_z_)
```

这是一个很小但值得修复的参数使用问题。

### 2.7 package 元数据未完善

多个 `package.xml` 中仍有：

```text
TODO: Package description
TODO: License declaration
```

交付前建议补齐。

### 2.8 部分参数和注释不完全一致

例如 LiDAR 分割 YAML 中有一些旧参数名，而代码实际读取的是另一组参数。

改进：

- 清理无效配置项。
- 给每个参数写实际生效说明。

## 3. 建议改进路线

### 阶段一：工程交付整理

优先级最高，风险最低：

1. 补全 package description/license。
2. 删除或注释不雅日志。
3. 清理无效 YAML 参数。
4. 将 `/dev/ttyUSB0` 改为 ROS 参数。
5. 增加 `/global_cone_map` 结构化输出。
6. 把 `gyro_factor` 接入动态噪声。

### 阶段二：融合精度提升

1. 使用畸变校正。
2. 加入全局一对一匹配。
3. 将 IoU、中心距离和类别置信度组合成匹配代价。
4. 对单目补漏改为地平面射线求交。
5. 对强制上色半径做自适应。

### 阶段三：SLAM 后端增强

1. Unknown 低权重入图。
2. 增加锥桶观测次数统计。
3. 发布协方差信息。
4. 增加地图保存/加载。
5. 增加离群点批量清理。
6. 循迹回环后做地图一致性修正。

### 阶段四：skidpad 专用策略

八字赛道需要考虑：

- 左右圆环。
- 入口和出口橙锥。
- 多圈重复经过。
- 对称结构导致的数据关联歧义。

建议单独实现：

- 入口检测。
- 圆环锥桶分组。
- 左右环状态机。
- 多圈统计稳定地图。

## 4. 汇报中如何表达不足

不要说“这个没做完”，可以说：

```text
当前工程已完成直线和循迹两类主要链路，并为 skidpad 预留了模式接口。八字绕环由于赛道拓扑和状态机更复杂，计划作为下一阶段专用策略实现。
```

对于 MarkerArray：

```text
当前 `/global_map` 主要服务 RViz 可视化。若对接规划层，下一步会增加结构化全局地图 topic，保留锥桶 id、颜色、位置和置信度。
```

对于 force match：

```text
当前使用中心距离容错策略提升外参轻微误差下的颜色召回，后续会替换为基于综合代价的一对一全局匹配。
```

## 5. 最终评价

当前工程的核心价值不在于某个单独算法特别复杂，而在于它把比赛可用的多模块链路打通了：

```text
数据输入
  -> 感知检测
  -> 多模态融合
  -> GNSS 位姿对齐
  -> 全局锥桶地图维护
  -> 可视化和端到端验证
```

对一个无人赛车工程来说，这比单点算法 demo 更重要。
