# 02 从传感器到融合输出

这一份文档只讲前半段。

起点是传感器原始数据。

终点是 `/perception/fusion/map`。

读完这一份，你应该能自己重写下面四个模块：

1. GNSS/INS 数据接入
2. YOLO 相机锥桶检测
3. LiDAR 锥桶分割
4. 相机和 LiDAR 融合

## 1. GNSS/INS 数据怎么进系统

核心文件：

- `gnss/cpp_pubsub/src/publisher_member_function.cpp`
- `gnss/gnss_ins_msg/msg/Gnssins64.msg`

### 1.1 节点启动方式

主函数在 `publisher_member_function.cpp:578-607`。

流程很直接：

1. 固定串口设备 `/dev/ttyUSB0`
2. 固定波特率 `460800`
3. `Stream::create_serial(...)`
4. 一直尝试连接
5. 连接成功后 `rclcpp::spin(std::make_shared<MinimalPublisher>())`

这里有一个很重要的实现细节。

`MinimalPublisher` 的构造函数里有一个 `while(1)` 串口读取循环，代码在 `publisher_member_function.cpp:266-416`。

主工作线程就卡在构造函数里，持续读串口，再直接发布消息。

这个写法很硬，但它能跑。

如果你自己重写，建议把串口读取放到：

- 独立线程
- 或定时器回调
- 或异步 IO

逻辑会更清楚。

### 1.2 数据帧格式怎么解

代码在 `publisher_member_function.cpp:202-399`。

关键点有 4 个：

1. 用 `findhead` 找帧头 `AA44AA45`
2. 单帧长度固定 `172` 字节
3. 用 `Calculate_Crc8` 验证前 171 字节
4. 按固定字节偏移解析字段

几个关键偏移：

- `+4` 纬度
- `+8` 经度
- `+12` 高程
- `+16` 东向速度
- `+18` 北向速度
- `+22` pitch
- `+24` roll
- `+26` yaw
- `+28` INS 状态
- `+29` GNSS week
- `+31` GNSS second

这些值都是原始整数，需要乘比例系数。

例子：

- 纬度、经度乘 `1e-7`
- 高度乘 `1e-3`
- 速度乘 `1e2 / 2^15`
- 姿态角乘 `180 / 2^15`

### 1.3 时间戳怎么来

代码在：

- `epoch2time`
- `gpst2time`
- `timeadd`
- `GPSTime2UTCTime`
- `parseGONGJI`

对应位置：

- `publisher_member_function.cpp:119-167`
- `publisher_member_function.cpp:495-503`

做法是：

1. 用 `gnss_week` 和 `gnss_second`
2. 先转 GPS 时间
3. 再减闰秒 `LEAPS=18`
4. 得到 UTC
5. 填到 ROS `header.stamp`

这个时间戳很重要。

后面 SLAM 的同步回调就靠它和感知消息对齐。

### 1.4 节点发了哪些话题

发布者定义在 `publisher_member_function.cpp:269-274`。

一共发 4 类东西。

| 话题 | 类型 | 用途 |
| --- | --- | --- |
| `gongji_gnss_ins` | `Gnssins` | 旧版 float 精度消息 |
| `gongji_gnss_ins_64` | `Gnssins64` | 主用高精度消息 |
| `imu` | `sensor_msgs/Imu` | IMU 标准消息 |
| `body_velocity` | `geometry_msgs/Vector3Stamped` | 车体系速度 |

SLAM 实际订阅的是：

- `gongji_gnss_ins_64`

### 1.5 `Gnssins64` 里最重要的字段

消息定义在 `Gnssins64.msg:1-30`。

SLAM 主要用：

- `latitude`
- `longitude`
- `vel_e`
- `vel_n`
- `pitch`
- `roll`
- `yaw`
- `imu_gyro_z`

这几个字段已经够建图了。

### 1.6 复现这个节点时要保留什么

最少要保留下面几点：

1. 经纬度和姿态角解析
2. GPS 周秒转 ROS 时间戳
3. 双精度经纬度
4. 东北速度
5. z 轴角速度

如果这些丢了，后面 SLAM 会很难复现得像原工程。

### 2.6 HSV 检查到底在干什么

相关代码在：

- `_expected_color`
- `_classify_hsv_color`
- `_update_hsv_stats`

位置：

- `yolo_detector.py:151-262`

1. 取检测框 ROI
2. 转 HSV
3. 用 H 通道中值估颜色
4. 和模型类别颜色做对比
5. 不一致就把调试框画成红色

这个工具很实用。因为锥桶场景里颜色是强语义信息。

### 3.5 锥桶怎么判

代码：

- `test_cone_segmentation.cpp:235-318`

每个 cluster 会走两层判断。

#### 第一层：几何尺寸初筛

代码：

- `268-270`

规则：

1. 高度在 `MIN_CONE_HEIGHT` 到 `MAX_CONE_HEIGHT`
2. 平面尺寸不能太大

#### 第二层：圆锥模型拟合

代码：

- `275-301`

方法：

1. 先估算法向
2. 用 `SACSegmentationFromNormals`
3. 模型设成 `SACMODEL_CONE`
4. 拟合成功后取锥顶
5. 再检查锥顶和几何中心偏差

关键设置：

- `setRadiusLimits`
- `setMinMaxOpeningAngle`
- `setAxis(Eigen::Vector3f(0, 0, 1))`
- `setEpsAngle(M_PI / 6)`

这说明它假设锥桶大体竖直。

### 3.6 拟合失败怎么办

代码：

- `303-310`

如果拟合失败，就走几何回退：

1. 用 `detectConeByGeometry` 再验一次
2. 通过就拿包围盒中心和最高点近似锥体
3. 还不通过就丢掉

这一步很实用。

真实点云里，很多小目标点数不够，直接靠圆锥拟合会丢目标。

### 4.5 3D 框怎么投到图像

函数：

- `project_3d_boxes_to_2d`

- `fs_fusion_box_math.cpp:8-84`

流程：

### 4.6 LiDAR 和相机怎么配对

函数：

- `fuse_measurements`

位置：

- `fs_fusion_box_math.cpp:97-169`

逻辑很直接。

对每个 LiDAR 投影框：

1. 取所有相机框
2. 算 IoU
3. 找最大 IoU
4. 超过阈值就拿这个相机框的类别颜色
5. 没过阈值就记成 `UNKNOWN`

输出是 `FusionResult`：

- `fused_cones`
- `unmatched_camera_indices`

这里的 `fused_cones` 已经有 LiDAR 的二维位置和相机的颜色。

### 4.7 远处纯视觉锥桶怎么补

函数：

- `recover_missing_cones`

位置：

- `fs_fusion_box_math.cpp:171-218`

这一步用来补“相机看到了，LiDAR 没看到”的目标。

做法很简单：

1. 对没匹配上的 2D 框
2. 假设锥桶真实高度 `0.30m`
3. 用相似三角形估深度
4. 在相机系求一个 3D 点
5. 用 `T_c2l` 变回 LiDAR 系
6. 颜色直接沿用 YOLO 类别

公式就在代码里：

```cpp
z_c = (fy * real_height) / det.bbox.size_y;
x_c = (u - cx) * z_c / fx;
```

这一步很工程化。

精度不算高，但它能把远距锥桶补回来。

### 4.8 “吸铁石”逻辑在干什么

代码：

- `fs_fusion_box_node.cpp:126-177`

这段逻辑很有工程味。

场景是：

- LiDAR 看到了近处目标
- 投影框 IoU 没对上
- 颜色掉成灰色 `UNKNOWN`

修复办法：

1. 把灰色锥桶中心再投影一次
2. 找最近的 2D 框中心
3. 如果像素距离小于 `force_match_radius`
4. 强制把颜色吸过去

这个方法不优雅，但很实用。

它能顶住轻微外参误差和小框偏差。

## 7. 这一段代码里最该记住的实现要点

### 7.1 颜色来自相机

LiDAR 分割本身不判颜色。

### 7.2 位置主要来自 LiDAR

融合输出里的 `x/y` 主要来自 LiDAR 的 3D 目标中心。

### 7.3 时间戳一定要传下去

感知消息如果换了时间戳，SLAM 同步就会漂。

### 7.4 外参矩阵是整个前端链的命门

`lidar_to_camera_matrix` 一旦偏大，IoU 会掉，吸铁石逻辑会更忙，颜色错误会增多。

### 7.5 融合层输出的是“当前帧局部锥桶观测”
