# 环境依赖与构建流程

## 1. 运行环境定位

工程目标环境是：

```text
ROS 2 Humble
Ubuntu 22.04 常见系统环境
Python 3.10
C++17/ament_cmake
```

同时需要机器学习依赖：

- PyTorch
- torchvision
- ultralytics
- OpenCV Python

为了避免 Conda 或全局 Python 污染 ROS 环境，工程提供了专门的 ROS + ML 虚拟环境脚本。

## 2. 关键脚本

```text
scripts/setup_ros_ml_venv.sh
scripts/activate_ros_ml.sh
scripts/build_ros_clean.sh
scripts/build_ros_venv.sh
scripts/install_ros_apt_deps_local.sh
scripts/install_csf_local.sh
```

## 3. Python 虚拟环境

创建或刷新：

```bash
./scripts/setup_ros_ml_venv.sh
```

该脚本创建：

```text
.venv_ros_ml
```

特点：

```text
python3 -m venv --system-site-packages
```

原因：

- ROS 2 Humble 的 Python 包通常安装在系统路径。
- `--system-site-packages` 可以让虚拟环境看到 ROS Python 包。
- ML 依赖安装在 venv 中，避免污染系统 Python。

## 4. setup_ros_ml_venv.sh 做了什么

主要步骤：

1. source `/opt/ros/humble/setup.bash`。
2. 创建 `.venv_ros_ml`。
3. 升级 pip/wheel。
4. 限制 `setuptools<80`，避免 ROS Python 包兼容问题。
5. 安装 PyTorch 和 torchvision。
6. 安装 numpy、opencv、pandas、polars、psutil、seaborn、tqdm。
7. 安装 ultralytics。
8. 验证导入：

```text
rclpy
cv2
numpy
torch
torchvision
ultralytics
```

默认安装 CPU PyTorch：

```text
TORCH_VARIANT=cpu
```

如果需要 CUDA 12.1：

```bash
TORCH_VARIANT=cu121 ./scripts/setup_ros_ml_venv.sh
```

## 5. 激活环境

```bash
source scripts/activate_ros_ml.sh
```

该脚本做了几件很关键的事：

1. 检查必须被 source，不能直接执行。
2. 清理 Conda 相关环境变量。
3. 设置 `PYTHONNOUSERSITE=1`。
4. source ROS 2。
5. 激活 `.venv_ros_ml`。
6. 加入本地 ROS 依赖前缀 `.ros_deps/opt/ros/humble`。
7. 加入本地 CSF 依赖 `.ros_deps/csf`。
8. 设置：

```text
ROS_VENV_DIR
ROS_PYTHON_EXECUTABLE
CSF_ROOT
```

## 6. 本地依赖 vision_msgs

融合节点依赖：

```text
vision_msgs
```

如果不能用 sudo apt 全局安装，工程提供：

```bash
./scripts/install_ros_apt_deps_local.sh ros-humble-vision-msgs
```

它会：

1. `apt-get download` 下载 deb。
2. `dpkg-deb -x` 解压到 `.ros_deps`。
3. 写入 local_setup 脚本。
4. 让 `activate_ros_ml.sh` 把该路径加入 AMENT/CMAKE/PYTHON/LD 路径。

## 7. 本地依赖 CSF

LiDAR 分割依赖：

```text
CSF.h
libCSF
```

安装脚本：

```bash
./scripts/install_csf_local.sh
```

它会：

1. clone 官方 CSF 仓库到 `.ros_deps/src/CSF`。
2. cmake 构建到 `.ros_deps/build/csf`。
3. 安装到 `.ros_deps/csf`。

构建 `test_cone_segmentation` 时，CMake 会查找：

```text
CSF_ROOT/include/CSF.h
CSF_ROOT/lib/libCSF.*
```

## 8. 构建脚本 build_ros_venv.sh

推荐构建：

```bash
./scripts/build_ros_venv.sh --event-handlers console_direct+
```

该脚本只是设置：

```text
ROS_VENV_DIR=.venv_ros_ml
ROS_PYTHON_EXECUTABLE=.venv_ros_ml/bin/python3
```

然后调用：

```bash
scripts/build_ros_clean.sh
```

## 9. build_ros_clean.sh 做了什么

关键逻辑：

1. 清理 Conda 环境变量。
2. 过滤 PATH 中的 conda 路径。
3. source ROS 2。
4. 加入本地 vision_msgs 和 CSF。
5. 验证 Python 包：

```python
import em
import numpy
import colcon_core
```

6. 设置默认 base paths：

```text
gnss/gnss_ins_msg
gnss/cpp_pubsub
slam
perception/src/cone_ws/src/cone_interfaces
perception/src/cone_ws/src/cone_detector
perception/src/cone_segmentation_test_3d/src/test_cone_segmentation
perception/src/fs_fusion_box
perception/src/run_perception
```

7. 执行 colcon：

```bash
python3 -m colcon build \
  --symlink-install \
  --cmake-clean-cache \
  --base-paths ... \
  --cmake-args -DPython3_EXECUTABLE=<venv_python>
```

## 10. 为什么指定 base paths

仓库里存在：

```text
perception/src/drd25_msgs
```

但它被 `COLCON_IGNORE` 禁用。构建脚本还主动指定 base paths，避免：

- 误构建重复包。
- 构建 build/install/log 目录。
- 引入无关代码。

## 13. 构建常见失败点

### 13.1 找不到 vision_msgs

解决：

```bash
./scripts/install_ros_apt_deps_local.sh ros-humble-vision-msgs
source scripts/activate_ros_ml.sh
```

### 13.2 找不到 CSF.h

解决：

```bash
./scripts/install_csf_local.sh
source scripts/activate_ros_ml.sh
```

### 13.3 Python 找不到 rclpy

原因：

- 没有 `--system-site-packages`。
- 没有 source ROS。
- Conda 污染。

解决：

```bash
source scripts/activate_ros_ml.sh
```

### 13.4 YOLO 节点找不到模型

检查：

```text
config/hardcoded_paths.ini
models.yolo_runtime
```

## 14. 工程目录体积提醒

当前工程中：

- `.venv_ros_ml` 体积较大，因为包含 PyTorch。
- `.ros_deps` 是本地依赖，不是源码核心。
- `build/`、`install/`、`log/` 是构建和运行产物。

交付源码时通常保留源码和脚本，不一定保留大体积虚拟环境。
