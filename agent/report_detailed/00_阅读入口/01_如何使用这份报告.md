# 如何使用这份报告

这份报告按“工程接手”的方式组织，而不是按“源码文件顺序”组织。原因很简单：真实复现一个系统时，你不会先去背每个文件，而是先弄清楚数据如何流动、节点如何连接、坐标如何统一、核心状态如何更新。

## 第一种用法：用于汇报

建议汇报叙事如下：

1. 需求背景：无人方程式赛道由彩色锥桶构成，规划控制需要可靠的全局锥桶地图。
2. 系统方案：相机负责颜色，LiDAR 负责空间，GNSS/INS 负责车辆全局位姿，SLAM 后端负责稳定建图。
3. 感知链路：YOLO 检测二维框，点云分割检测三维框，融合层输出带颜色的锥桶地图。
4. SLAM 链路：把每帧局部锥桶观测转换到全局坐标，做数据关联、卡尔曼更新、存在评分、稳定发布。
5. 工程验证：一键回放 rosbag，关键话题全部出数，最终 `/global_map` 非空。
6. 改进方向：八字模式、结构化地图发布、融合容错策略正规化、参数收敛和自动标定。

## 第二种用法：用于复现

如果要从零写一个最小可运行版本，按这个顺序实现：

1. 定义消息：`Cone`、`Map`、`Gnssins64`、`ConeArray`、`ThreeDConeArray`。
2. 写 YOLO 节点：订阅图像，发布二维锥桶框和颜色。
3. 写 LiDAR 节点：订阅点云，地面分割、聚类、筛选，发布三维锥桶中心和尺寸。
4. 写融合节点：投影 3D 框到图像，IoU 匹配，输出 `drd25_msgs/Map`。
5. 写 SLAM 节点：订阅 GNSS/INS 和融合地图，坐标转换，数据关联，卡尔曼更新，发布全局地图。
6. 写 launch：把节点和话题 remap 串起来。
7. 写运行脚本：统一环境、启动节点、播放 rosbag、检查关键 topic。

## 第三种用法：用于代码审查

每章都包含：

- 对应源码路径。
- 输入输出。
- 参数。
- 代码执行流程。
- 核心公式或伪代码。
- 复现时需要注意的坑。

你可以把每章当成一份 code walkthrough 清单。

## 本报告中的命名约定

为了减少歧义，本报告采用以下称呼：

- “感知地图”：指融合节点输出的 `/perception/fusion/map`，类型为 `drd25_msgs/msg/Map`。
- “全局地图”：指 SLAM 节点内部维护的 `cone_map_`。
- “发布地图”：指 SLAM 发布到 `/global_map` 的 `visualization_msgs/msg/MarkerArray`。
- “车辆坐标系”：通常指 ROS 风格车体系，`x` 前、`y` 左、`z` 上。
- “地图坐标系”：SLAM 的 `map` frame，局部 ENU，`x` 近似东向、`y` 近似北向。
- “LiDAR 坐标系”：融合输出锥桶位置所在的局部传感器坐标系，由外参变换到车体系。

## 重要提醒

当前仓库有一个被禁用的重复消息包：

```text
perception/src/drd25_msgs/COLCON_IGNORE
```

因此工程中 `drd25_msgs` 的有效来源是：

```text
slam/drd25_msgs
```

复现时不要同时启用两个 `drd25_msgs`，否则会产生消息定义冲突。
