# 系统目标与总体架构

## 1. 工程目标

本工程目标是为无人方程式赛车构建全局锥桶地图。地图中的锥桶需要同时具有：

- 全局坐标位置。
- 锥桶颜色语义。
- 稳定性判断。
- 可视化输出能力。

赛车赛道通常由锥桶组成。规划控制模块关心的是：

- 左右边界在哪里。
- 颜色是否可靠。
- 远处锥桶是否可信。
- 地图是否会闪烁、漂移、重复。
- 第一圈探索后是否能固定地图用于后续快速通过。

因此工程没有走“只靠视觉”或“只靠 LiDAR”的路线，而是将多个传感器的优势组合起来：

```text
Camera：识别颜色和 2D 位置。
LiDAR：提供锥桶局部空间坐标。
GNSS/INS：提供车辆全局位姿。
SLAM 后端：维护稳定全局锥桶地图。
```

## 2. 为什么说它是语义锥桶建图

传统 SLAM 通常强调“机器人同时定位和建图”。当前工程的定位已经由 GNSS/INS 提供，SLAM 节点不估计车辆位姿，它主要负责：

- 把感知输出的局部锥桶观测变换到全局地图。
- 多帧融合锥桶位置。
- 去除噪点和重复点。
- 用颜色和统计评分维护稳定锥桶。
- 依据赛道模式执行锁图和回环逻辑。

因此更准确的技术描述是：

```text
GNSS/INS 位姿驱动的多传感器语义锥桶建图系统。
```

## 4. 每个模块的职责

### 4.1 相机 YOLO 检测

源码：

```text
perception/src/cone_ws/src/cone_detector/cone_detector/yolo_detector.py
```

职责：

- 订阅相机图像。
- 加载 YOLOv8 权重。
- 检测锥桶 2D bounding box。
- 根据模型类别名映射锥桶颜色。
- 发布自定义 `cone_interfaces/msg/ConeArray`。

输出：

```text
/yolo/cones
/yolo/debug_image
```

### 4.2 LiDAR 点云分割

源码：

```text
perception/src/cone_segmentation_test_3d/src/test_cone_segmentation/src/test_cone_segmentation.cpp
```

职责：

- 订阅 LiDAR 点云。
- 做 ROI 过滤、体素降采样、地面分割。
- 对障碍物点云做欧几里得聚类。
- 对每个 cluster 做锥桶几何筛选和可选圆锥拟合。
- 发布自定义 `test_cone_segmentation/msg/ThreeDConeArray`。

输出：

```text
/cone_detection_custom
/cone_bboxes
/cone_markers
/debug/ground
/debug/obstacles
```

### 4.3 LiDAR-Camera 融合

源码：

```text
perception/src/fs_fusion_box/src/fs_fusion_box_node.cpp
perception/src/fs_fusion_box/src/fs_fusion_box_math.cpp
```

职责：

- 同步 LiDAR 锥桶和相机锥桶。
- 将自定义消息转换为 `vision_msgs` 风格内部结构。
- 用外参和内参将 3D 框投影到相机画面。
- 通过 IoU 匹配 3D 投影框与 YOLO 2D 框。
- 输出带颜色的锥桶局部地图。

输出：

```text
/perception/fusion/map
/fusion/markers
```

### 4.4 GNSS/INS 接入

源码：

```text
gnss/cpp_pubsub/src/publisher_member_function.cpp
gnss/cpp_pubsub/src/serial_stream.cpp
```

职责：

- 从 `/dev/ttyUSB0` 读取惯导串口数据。
- 解析二进制帧。
- 计算 ROS 时间戳。
- 发布 GNSS/INS 自定义消息、IMU、车体系速度。

输出：

```text
/gongji_gnss_ins
/gongji_gnss_ins_64
/imu
/body_velocity
```

离线演示时，rosbag 直接提供 `/gongji_gnss_ins_64`，可以不运行串口节点。

### 4.5 SLAM 后端

源码：

```text
slam/slam/src/slam_node.cpp
slam/slam/src/loop_closure_detector.cpp
```

职责：

- 订阅 GNSS/INS 和融合地图。
- 用 PROJ 将经纬度转 UTM。
- 用第一帧 GNSS 建立局部地图原点。
- 发布车辆 TF、Odometry、Path。
- 把局部锥桶观测转换到全局地图。
- 进行数据关联、卡尔曼更新、存在概率维护。
- 发布稳定全局锥桶地图。
- 在循迹模式中执行回环检测和锁图。

输出：

```text
/global_map
/vehicle_path
/vehicle_odom
TF: map -> base_link
```

## 5. 关键设计思想

### 5.1 位置和语义分工

相机不直接承担三维定位主责，LiDAR 不直接承担颜色识别主责。

```text
LiDAR 给位置。
Camera 给颜色。
Fusion 做颜色赋值。
SLAM 做时序稳定。
```

这样设计的好处是每个传感器只做自己擅长的事情。

### 5.2 GNSS/INS 直接提供车辆位姿

SLAM 节点不做 scan matching，也不做视觉里程计。车辆位姿来自 `/gongji_gnss_ins_64`：

- `latitude`
- `longitude`
- `roll`
- `pitch`
- `yaw`
- `vel_e`
- `vel_n`
- `imu_gyro_z`

然后用 PROJ 转换为地图坐标。

### 5.3 地图对象是锥桶，不是点云

SLAM 后端维护的是：

```cpp
std::vector<GlobalCone> cone_map_;
```

每个 `GlobalCone` 包含：

- `id`
- `pos`
- `P`
- `r, g, b`
- `is_stable`
- `existence_score`
- `matched_this_frame`
- `type`

也就是说，地图的基本单位是“锥桶实体”，不是点云点。

### 5.4 多层防噪策略

工程使用多层机制避免噪点污染地图：

- 感知层尺寸筛选。
- 融合层 IoU 匹配。
- SLAM 层颜色门控。
- SLAM 层欧氏距离粗筛。
- SLAM 层马氏距离精筛。
- SLAM 层 Kalman 滤波。
- SLAM 层 hit/miss 存在评分。
- SLAM 层稳定阈值发布。
- 循迹模式第一圈后锁图。

## 6. 当前实现边界

当前工程已经实现：

- 相机 YOLO 检测。
- LiDAR 点云锥桶分割。
- LiDAR-Camera 融合。
- GNSS/INS 到局部地图坐标。
- 直线建图。
- 循迹建图。
- 几何回环检测。
- 回环后锁图。
- rosbag 一键验证。

当前工程未完整实现：

- 八字绕环 `skidpad` 的专门建图策略。
- 结构化全局地图数据消息发布，当前 `/global_map` 是可视化 Marker。
- 严格的图优化后端。
- 基于地图一致性的全局位姿修正。
- 多传感器自动标定。
