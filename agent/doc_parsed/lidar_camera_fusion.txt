           lidar识别出3d的框，包含x，y，长，宽，高；相机识别出2d的框，包含x，y，长，宽，cone类型

           通过算法将3d的框投影到2d的画面上，这时对应的位置会有2个框。这两个框的重叠部分的面积与未重叠
           的部分的面积的比值计算为置信度。
           置信度高：保留，将2d框中的cone类型赋到3d的框上，然后传出x，y，cone类型的话题
           置信度低：标上一个unkown，只要表示这里可能有一个锥筒就可，通过下一帧或几帧来辅助确认
           如果yolo识别出来的锥筒数小于雷达的，以yolo识别出的为主，对多出来的雷达识别框打上unkown
           如果yolo识别出来的锥筒数大于雷达的，则通过画面反推3d的位置，把雷达少检测的锥筒的位置推断出
           来（精度很低）

           本功能包负责接收激光雷达 (LiDAR) 的 3D 障碍物检测框 和 相机 (Camera) 的 2D 识别框，通过时空同
           步与多模态融合算法，输出带有颜色分类和精确三维坐标的锥筒地图，供下游规划控制模块使用。
           2.   算法流程 (Pipeline)
               1. 时间同步 (Time Synchronization)：
                    使用 ﻿message_filters﻿ 的近似时间同步策略 (ApproximateTime)，确保雷达和相机数据在
                     ○


                    时间戳上对齐（默认窗口 0.1s）。
               2. 空间投影 (Spatial Projection)：

                    利用外参矩阵（$T_{lidar \to camera}$）和相机内参（$K$），将雷达 3D 框的 8 个角点投影到
                     ○


                    相机 2D 图像平面，生成“雷达投影框”。
               3. 数据关联 (Data Association)：

                    计算“雷达投影框”与“YOLO 2D 框”的 IoU (交并比)。
                     ○



                    若 IoU > 阈值，判定为同一物体，将相机的颜色属性赋予雷达的空间坐标。
                     ○



               4. 补漏与容错 (Recovery & Fallback)：

                    高置信度：雷达相机均检测到 -> 融合输出（位置准、颜色准）。
                     ○



                    雷达单源：相机未检出或在视野外 -> 输出 Unknown 颜色锥筒（保位置）。
                     ○



                    相机单源：雷达漏检（通常在远端） -> 基于单目几何原理（假设地面平坦）反推 3D 坐标（保
                     ○


                    召回率）。

           fs_fusion_box: FS Driverless                                感知融合系统
           项目简介
                 是无人方程式赛车（Formula Student Driverless）感知系统的核心融合节点。
           fs_fusion_box

           该节点采用多模态融合架构，接收来自激光雷达的 3D 聚类结果 和来自相机的 2D YOLO 识别框，通过
           时空同步与投影匹配算法，输出带有精确三维坐标与颜色分类的锥筒地图，供下游 SLAM 和规划控制模
           块使用。


           1.       核心特性 (Key Features)
               双适配器架构 (Dual-Adapter Pattern)
                ●



           内部集成了数据转换层，无缝对接 ﻿test_cone_segmentation﻿ (3D) 和 ﻿cone_interfaces﻿
           (2D) 自定义消息，同时保持核心算法库基于标准 ﻿        vision_msgs﻿ 的纯净性。

               鲁棒的时空同步
                ●



           使用 ﻿ApproximateTime﻿ 策略自动对齐雷达与相机的时间戳（默认容差 0.1s），确保高速运动下的数
           据一致性。
               智能补漏 (Recovery Mechanism)
                ●



           当雷达漏检（如远距离）但相机检出时，基于单目几何原理反推 3D 坐标，最大化感知召回率。
               可视化调试系统
                ●



           内置 Rviz MarkerArray 发布器，实时显示融合后的锥筒位置、颜色及类别，支持“所见即所得”的调试。

           对于这种问题，我思考了几种可能：
           通过观察rviz2上的表现，我首先怀疑是延迟的问题。
           对于延迟问题，我做出了如下的改进：
           （1）将融合和可视化这两个分为两个单独的进程，同时启动来降低延迟
           （2）原来融合和可视化这两个进程全放在callback里面，复杂计算、文件 I/O、网络请求等阻塞操作会破
           坏时间确定性，一旦有一个进程卡住，就会阻塞后面所有的进程。所以我修改了代码，让callback仅完成
           一些数据的处理，以保证实时性。

           然而以上的改动确实降低了延迟，但是还是有位置差别
           所以第二个想到的是雷达识别时的w（偏转角），因为雷达的可视化使用了w,而我的可视化没有使用w,所
           以我认为问题可能出现在这上面。对此，我让雷达传入的信息包含了他检测到的w,然后我的可视化再应
           用这个w来改进他的可视化逻辑。但是最后呈现的效果还是不是很好。
           随后我选择了全部按照雷达的可视化逻辑，但是最后还是这样。

           我输出了坐标和坐标系，最后得到的都是正确的。所以也不是这个问题。

           随后我认为是我的匹配机制出现了问题，所以我修改了对于雷达每一帧的匹配机制。
           原来的匹配机制： ApproximateTime 近似时间同步 ，即：
             1. 雷达和相机消息分别进入各自的队列
             2. 同步器在两个队列里，寻找 “时间戳最接近” 的一对消息

             3. 如果时间差 < 你设置的 sync_window（比如 0.1 秒），就认为匹配成功，触发回调



           经过修改之后的匹配机制： LatestTime 机制
           不管是雷达来新消息，还是相机来新消息，立刻触发配对。配对规则只有一条：谁最新，就用
           谁。
           其实这并不能解决问题，但是关键在于我多做了一件事：打印出了匹配到的相机和雷达之间时间戳
           的差。结果我惊讶地发现这两者之间相差了将近750ms。
