# 无人驾驶方程式赛车实时锥桶建图模块工作介绍

## 1. 工作概述

我在该项目中主要负责的是 **GNSS/INS 辅助的实时锥桶建图模块**。

这个模块位于感知融合之后、规划控制之前，主要作用是把前端感知输出的局部锥桶观测，结合车辆的 GNSS/INS 位姿，转换为稳定的全局锥桶地图，为后续路径规划提供可靠的赛道边界信息。

更具体地说，我的工作不是重新实现整套无人车系统，而是聚焦在建图后端：如何把一帧帧局部锥桶观测稳定地写入全局地图，并尽量避免误检、漏检或感知异常污染长期地图。

## 2. 项目背景

在无人驾驶大学生方程式赛车中，赛道通常由蓝色、黄色、红色等锥桶构成。车辆需要实时感知锥桶位置，并根据左右边界生成可行驶中心线。

前端感知只能给出当前车辆坐标系下的局部锥桶观测，而规划模块需要的是稳定、连续、全局一致的赛道地图。因此，中间必须有一个实时建图模块，完成以下任务：

- 将局部锥桶坐标转换到全局地图坐标系；
- 判断当前观测和历史锥桶是否对应同一个物体；
- 对锥桶位置进行滤波更新，降低单帧噪声影响；
- 区分稳定锥桶和低可信候选锥桶；
- 在感知异常时避免错误观测写入长期地图。

这一部分直接影响后续规划控制的输入质量。如果地图中出现重复锥桶、错误锥桶或位置跳变，规划结果就可能偏离真实赛道边界。

## 3. 我的主要工作

### 3.1 GNSS/INS 辅助的全局坐标转换

建图模块订阅 GNSS/INS 消息 `/gongji_gnss_ins_64` 和感知融合结果 `/perception/fusion/map`。

我在建图过程中使用 GNSS/INS 提供的经纬度、航向角、速度和角速度信息，构建车辆在全局地图中的位姿。具体流程包括：

1. 将 GNSS 经纬度从 WGS84 坐标系转换为 UTM 平面坐标；
2. 以第一帧 GNSS 位置作为局部地图原点，避免全局坐标数值过大；
3. 根据 roll、pitch、yaw 构建车辆姿态旋转矩阵；
4. 结合 LiDAR 到车体的外参矩阵，将局部锥桶坐标投影到全局地图坐标系。

核心转换关系可以概括为：

```text
p_global = T_vehicle_to_map * T_lidar_to_vehicle * p_lidar
```

这样，每一帧感知输出的局部锥桶都可以被放到统一的全局地图中。

### 3.2 锥桶数据关联与位置更新

局部观测投影到全局地图后，关键问题是判断它是一个新锥桶，还是地图中已有锥桶的再次观测。

我实现的关联策略主要分为三步：

1. **颜色约束**：只有颜色类别一致的锥桶才参与匹配，避免蓝桶、黄桶之间错误关联；
2. **欧氏距离粗筛**：先根据空间距离过滤明显不可能匹配的锥桶；
3. **马氏距离精筛**：结合锥桶位置协方差和当前观测噪声，选择最可信的匹配对象。

匹配成功后，使用二维 Kalman 更新锥桶位置和协方差：

```text
P = P + Q
S = P + R
K = P * S^-1
pos = pos + K * (z - pos)
P = (I - K) * P
```

其中，`pos` 是地图中锥桶的位置，`P` 是位置协方差，`z` 是当前观测，`R` 是观测噪声。

为了更贴近赛车工况，观测噪声不是固定值，而是会根据观测距离、车辆速度和角速度动态调整。距离越远、转弯越剧烈，观测不确定性越大，对地图更新的影响也会相应降低。

### 3.3 候选锥桶、稳定锥桶与地图维护

为了避免单帧误检直接进入全局地图，我设计了锥桶存在分数机制。

每个锥桶都有一个 `existence_score`：

- 被当前帧成功观测到时加分；
- 在 LiDAR 视野内但没有被观测到时减分；
- 分数超过稳定阈值后，才作为稳定锥桶发布到 `/global_map`；
- 分数低于删除阈值后，从地图中移除。

因此，系统不会把所有观测都直接发布给规划，而是将地图分成三层：

```text
/global_map                    稳定锥桶地图
/mapping/candidate_cones       候选锥桶层
/mapping/rejected_observations 被拒绝观测层
```

这种分层方式的好处是：规划模块只消费稳定锥桶，调试和回放分析仍然可以看到候选锥桶和被拒绝观测，方便判断地图质量。

### 3.4 新锥桶创建约束

对于未匹配上的观测，系统不会立即无条件创建新锥桶，而是先判断它是否位于合理的车辆行驶区域内。

我在新锥桶初始化时加入了基于阿克曼运动模型的 ROI 约束：

- 直线行驶时，使用车辆前方矩形通道；
- 转弯时，根据车速和角速度估计转弯半径，使用弯道管状区域；
- 不在 ROI 内的观测会被拒绝进入地图。

这样可以减少路边杂点、误检目标或不合理观测进入全局地图。

### 3.5 风险感知地图写入门控

比赛场景中，感知模块可能出现相机模糊、时间同步偏差、融合标定漂移、学习检测器输出异常等问题。如果这类观测被直接写入地图，会造成长期地图污染。

因此，我在建图模块中加入了风险感知地图写入门控。该门控会消费系统健康状态和感知失效状态，根据风险等级调整地图写入策略：

```text
open                    正常写入
monitor_only            只监控，不明显干预
downweight_observations 降低观测写入权重
freeze_new_landmarks    冻结新锥桶创建
```

当系统处于 degraded 状态时，已有锥桶仍然可以被低权重更新；当风险进一步升高时，系统会停止创建新锥桶，避免短时间的错误感知变成长期地图中的错误 landmark。

这部分是我认为该模块最有价值的地方：建图不只是追求“看到更多锥桶”，而是要判断“哪些观测不应该写进长期地图”。

## 4. 技术路线总结

整个建图模块可以概括为如下流程：

```text
GNSS/INS 位姿
    |
    | 经纬度转 UTM，构建车辆全局位姿
    v
融合锥桶观测 /perception/fusion/map
    |
    | 局部坐标转全局坐标
    v
颜色约束 + 欧氏距离粗筛 + 马氏距离精筛
    |
    | 匹配成功
    v
二维 Kalman 更新位置和协方差
    |
    | 匹配失败
    v
Ackermann ROI 判断是否允许新建锥桶
    |
    v
存在分数维护：命中加分，漏检减分
    |
    v
稳定地图 / 候选地图 / 拒绝观测层分层发布
    |
    v
风险门控保护长期地图
```

## 5. 实验与效果

在已有回放测试中，建图模块能够持续输出车辆轨迹、里程计和全局锥桶地图。

典型结果包括：

- PointPillars 后端下，`/global_map` 发布频率约为 **5.03 Hz**；
- 建图回调本身开销较低，`sync_callback_ms` 平均约 **0.32 ms**，P95 约 **0.47 ms**；
- 稳定锥桶地图中重复锥桶对为 **0**；
- 风险门控 A/B 测试中，稳定锥桶数量保持 **20/20**；
- 候选残留由 **11 降至 7**；
- 地图稳定性分数由 **0.545 提升至 0.721**。

这些结果说明，风险门控并不是简单减少地图数量，而是在保持稳定地图规模的同时，降低低可信候选残留，提高地图自一致性。

需要说明的是，当前指标主要是回放自一致性指标，还不是带人工标注真值的绝对地图精度评估。后续如果补充测绘锥桶真值或人工标注地图，可以进一步评估绝对位置误差。

## 6. 汇报时可以这样表述

如果在项目汇报中介绍我的工作，可以这样说：

> 我主要负责实时锥桶建图后端。前端感知会输出车辆坐标系下的局部锥桶，我的模块利用 GNSS/INS 位姿把它们转换到 UTM 全局坐标系中，然后通过颜色约束、距离筛选和马氏距离进行数据关联。匹配成功的锥桶用二维 Kalman 滤波更新位置和协方差，未匹配的观测则需要通过阿克曼 ROI 检查后才允许创建新锥桶。为了避免单帧误检污染地图，我还加入了存在分数机制，只有多次稳定观测到的锥桶才发布到 `/global_map`。此外，我设计了风险感知地图写入门控，在感知异常时降低观测权重或冻结新锥桶创建，从而保护后续规划使用的长期地图。

## 7. 面试中可以展开的技术点

### 7.1 为什么不用单帧锥桶直接规划？

单帧感知容易受到遮挡、漏检、误检和颜色识别错误影响。如果直接把单帧结果给规划，路径会抖动，而且错误观测可能导致中心线生成异常。建图模块通过时间上的多帧融合，把瞬时观测转化为更稳定的全局地图。

### 7.2 为什么用马氏距离而不是只用欧氏距离？

欧氏距离只考虑空间距离，不考虑不确定性。马氏距离会结合已有锥桶协方差和当前观测噪声，因此更适合处理不同距离、不同运动状态下观测可信度不同的问题。

### 7.3 为什么要有候选锥桶层？

如果所有新观测都直接进入稳定地图，误检会污染规划输入。候选层相当于一个缓冲区，只有经过多帧确认、存在分数足够高的锥桶才会进入稳定地图。

### 7.4 风险门控解决什么问题？

它解决的是“感知异常是否应该改变长期地图”的问题。感知模块短时间异常不可避免，但建图模块不能把异常结果永久写入地图。风险门控通过降权或冻结新锥桶创建，降低错误观测对长期地图的影响。

### 7.5 这个模块和传统 SLAM 有什么不同？

传统 SLAM 通常同时估计车辆轨迹和环境地图。本项目中车辆位姿主要由 GNSS/INS 提供，建图模块重点是构建稳定的语义锥桶地图。因此它更准确地说是 GNSS/INS 辅助的实时语义建图，而不是纯 LiDAR SLAM。

## 8. 工作边界说明

为了表述准确，我的主要工作聚焦在建图后端和地图写入策略：

- 前端 YOLO、PointPillars、PCL 聚类等感知模块不是我完整独立完成的部分；
- 我主要承接感知融合输出之后的地图更新、地图维护和风险门控；
- 当前已有结果能够证明链路实时性、地图自一致性和门控收益；
- 绝对定位/建图精度仍需要后续加入真值地图或参考轨迹进行评估。

这样的表述既能突出工作价值，也避免把整个系统都包装成个人独立完成。
